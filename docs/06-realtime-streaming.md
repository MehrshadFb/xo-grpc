# Realtime Streaming

Realtime updates are implemented with gRPC server-side streaming.

Public endpoint:

```proto
rpc WatchGame(WatchGameRequest) returns (stream GameEvent);
```

Internal event hub:

```text
internal/realtime
```

## Why Server-Side Streaming

Tic-Tac-Toe needs both players to see updates quickly:

- opponent joined
- move made
- game over
- rematch requested
- next round started

Polling would work, but it creates unnecessary repeated requests. Server-side streaming lets the client open one stream and receive events as they happen.

## Hub Design

Location: `internal/realtime/hub.go`

The hub keeps subscribers grouped by game id:

```go
map[string]map[Subscriber]struct{}
```

Where:

```go
type Subscriber chan Event
```

Each watcher gets a private buffered channel.

## Subscribe Flow

```text
WatchGame request
  |
  |-- validate player token through GameService.GetState
  |-- optionally send initial snapshot
  |-- subscribe to hub for game id
  |-- wait for events or disconnect
  v
stream GameEvent messages
```

On disconnect, the handler unsubscribes and closes the channel.

## Event Types

Internal event types:

```text
EventTypePlayerJoined
EventTypeMoveMade
EventTypeGameOver
EventTypeRematchRequested
EventTypeRoundStarted
```

Proto event types:

```text
GAME_EVENT_TYPE_STATE_SNAPSHOT
GAME_EVENT_TYPE_PLAYER_JOINED
GAME_EVENT_TYPE_MOVE_MADE
GAME_EVENT_TYPE_GAME_OVER
GAME_EVENT_TYPE_REMATCH_REQUESTED
GAME_EVENT_TYPE_ROUND_STARTED
```

`STATE_SNAPSHOT` is generated by the stream handler, not published by the hub.

## Full-State Events

Every streamed event includes the full `GameState`.

This is a deliberate design choice. It means the client does not need to apply patches or reconstruct state from event history. It can simply replace its local state with the event state.

Tradeoff:

- simpler client logic
- slightly larger messages

For a 3x3 board, the larger message size is not a problem.

## Resume Logic

Request:

```proto
message WatchGameRequest {
  string game_id = 1;
  string player_token = 2;
  int64 after_version = 3;
}
```

Behavior:

```text
if current version > after_version:
  send STATE_SNAPSHOT immediately
else:
  skip snapshot and wait for future events
```

This helps clients reconnect without receiving the same state twice.

Example:

1. Client has version `4`.
2. Client reconnects with `after_version = 4`.
3. Server sees current version is still `4`.
4. Server does not send a snapshot.
5. Client receives only future updates.

If the server has version `5`, it sends version `5` as a snapshot.

## Slow Subscribers

Each subscriber channel has buffer size `8`.

Publishing uses a non-blocking send:

```go
select {
case sub <- event:
default:
    // skip slow subscriber
}
```

This means one slow/disconnected client cannot block game updates for everyone else.

Tradeoff:

- protects the server from blocking
- a very slow client may miss an event

Because the client can reconnect with `after_version`, it can recover by asking for the latest state.

## Active Stream Metrics

`WatchGame` increments:

```text
xo_active_watch_streams
```

When the stream ends, it decrements the gauge.

This tells you how many active realtime watchers the backend currently has.

## Scaling Limitation

The hub is in memory. That means it works only inside one backend process.

Current deployment is one backend instance, so this is fine.

If the backend is scaled horizontally later:

```text
client A connected to backend 1
client B connected to backend 2
```

An event published in backend 1 would not automatically reach backend 2.

To support multiple backend replicas, replace or extend the hub with shared pub/sub:

- Redis Pub/Sub
- NATS
- PostgreSQL `LISTEN/NOTIFY`
- Kafka or another event log

For this project, the in-process hub is the right level of complexity.
